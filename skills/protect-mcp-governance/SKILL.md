# protect-mcp Governance

Cryptographic receipt signing and Cedar policy enforcement for AI agent tool calls. Every tool invocation gets an Ed25519 signed receipt and a policy decision (allow/deny/rate-limit) before execution.

## When to Activate

- Setting up agent governance or compliance controls
- Adding audit trails to Claude Code or MCP tool calls
- Writing Cedar policies to restrict which tools agents can use
- Migrating from shadow mode (log-only) to enforcement
- Verifying receipts offline for audit or incident response
- Integrating with enterprise observability (OpenTelemetry)

## Quick Start

### Claude Code Hooks + MCP Server

```bash
# Initialize hooks in your project (writes .claude/settings.local.json)
npx protect-mcp init-hooks

# Start the governance server (shadow mode by default)
npx protect-mcp serve
```

This registers PreToolUse and PostToolUse hooks that route every tool call through the protect-mcp gateway. Shadow mode logs decisions without blocking, so you can observe before enforcing.

### Enable Enforcement

```bash
# Switch from shadow to enforcement mode
npx protect-mcp serve --enforce
```

In enforcement mode, tool calls that violate policy are blocked with an explanation before execution.

## Cedar Policy Authoring

protect-mcp uses Cedar (the same policy engine behind AWS Verified Permissions) for authorization decisions. Place `.cedar` files in your project and they are auto-detected.

### Example: Allow reads, block dangerous writes

```cedar
// policies/default.cedar

// Allow all read operations
permit(
  principal,
  action == Action::"tool_call",
  resource
) when {
  resource.tool_name in ["Read", "Glob", "Grep"]
};

// Block destructive bash commands
forbid(
  principal,
  action == Action::"tool_call",
  resource
) when {
  resource.tool_name == "Bash" &&
  resource.command like "*rm -rf*"
};
```

### Example: Rate-limit by tool

```cedar
// Allow Bash but cap at 50 calls per session
permit(
  principal,
  action == Action::"tool_call",
  resource
) when {
  resource.tool_name == "Bash" &&
  context.session_tool_count < 50
};
```

### Policy file locations

```
your-project/
  policies/
    cedar/
      default.cedar    # Auto-detected by protect-mcp
      production.cedar  # Additional policy files
  .scopeblind/
    config.json        # Generated by init-hooks
    keypair.json       # Ed25519 signing key (gitignored)
```

## Deployment Modes

| Mode | Behavior | Use Case |
|------|----------|----------|
| **Shadow** (default) | Logs decisions, never blocks | Safe adoption, policy tuning |
| **Enforce** | Blocks policy violations | Production governance |
| **Proxy** | Wraps any MCP server | Add governance to existing servers |

### Shadow Mode Workflow

1. Run `npx protect-mcp init-hooks` in your project
2. Start the server: `npx protect-mcp serve`
3. Use Claude Code normally; all tool calls are logged with receipts
4. Review decisions: `npx protect-mcp status`
5. Tune Cedar policies based on observed patterns
6. Switch to enforce: `npx protect-mcp serve --enforce`

## Receipts and Verification

Every tool call produces an Ed25519 signed receipt containing:
- Tool name, input hash, timestamp
- Policy decision (allow/deny/rate-limit)
- Session and agent identifiers
- Causal chain linking receipts in sequence

### Verify a receipt offline

```bash
# Verify a single receipt
npx @veritasacta/verify receipt.json

# Verify without trusting any server (issuer-blind)
npx @veritasacta/verify --offline receipt.json
```

### View receipt log

```bash
# Show recent receipts
npx protect-mcp receipts --last 20

# Daily digest
npx protect-mcp digest --today

# Generate compliance report
npx protect-mcp report --period 7d --format md
```

## MCP Server Configuration

To add protect-mcp as a governance layer in your `.mcp.json`:

```json
{
  "mcpServers": {
    "protect-mcp": {
      "command": "npx",
      "args": ["-y", "protect-mcp", "serve"],
      "description": "Agent governance — Cedar policy enforcement + Ed25519 signed receipts for every tool call"
    }
  }
}
```

### Wrapping an existing MCP server

```json
{
  "mcpServers": {
    "github-governed": {
      "command": "npx",
      "args": [
        "-y", "protect-mcp", "--enforce",
        "--", "npx", "-y", "@modelcontextprotocol/server-github"
      ],
      "description": "GitHub MCP server wrapped with protect-mcp governance"
    }
  }
}
```

## Integration with Hooks

protect-mcp works alongside existing ECC hooks. The hooks fire in order:

```
PreToolUse hooks (ECC) -> protect-mcp policy check -> Tool executes -> PostToolUse hooks (ECC) -> Receipt signed
```

### Hook configuration (auto-generated by init-hooks)

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "npx protect-mcp check-pre"
          }
        ],
        "description": "protect-mcp: Cedar policy check before tool execution"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "npx protect-mcp check-post",
            "timeout": 10
          }
        ],
        "description": "protect-mcp: Sign receipt after tool execution"
      }
    ]
  }
}
```

## Anti-Patterns

### Do not hardcode policy in CLAUDE.md

```markdown
<!-- WRONG: Policy as natural language in CLAUDE.md -->
Never use the Bash tool to run rm commands.
```

Use Cedar policies instead. Natural language instructions can be ignored or overridden; Cedar policies are enforced programmatically.

### Do not skip shadow mode

Going straight to enforcement without observing traffic risks blocking legitimate tool calls. Always start in shadow mode, review the decision log, then switch to enforce.

### Do not commit signing keys

```gitignore
# .gitignore
.scopeblind/keypair.json
```

The keypair is generated locally by `init-hooks`. It should never be committed. The public key is embedded in receipts for verification.
