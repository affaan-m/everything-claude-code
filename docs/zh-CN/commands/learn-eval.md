# /learn-eval - 提取、评估、然后保存

扩展 `/learn`，在编写任何技能文件之前，加入质量门控、保存位置决策和知识放置意识。

## 提取内容

寻找：

1. **错误解决模式** — 根本原因 + 修复方法 + 可重用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方法** — 库的怪癖、API 限制、特定版本的修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

1. 回顾会话，寻找可提取的模式

2. 识别最有价值/可重用的见解

3. **确定保存位置：**
   * 提问："这个模式在其他项目中会有用吗？"
   * **全局** (`~/.claude/skills/learned/`)：可在 2 个以上项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技术等）
   * **项目** (当前项目中的 `.claude/skills/learned/`)：项目特定的知识（特定配置文件的怪癖、项目特定的架构决策等）
   * 不确定时，选择全局（将全局 → 项目移动比反向操作更容易）

4. 使用此格式起草技能文件：

```markdown
---
name: pattern-name
description: "Under 130 characters"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取日期：** [日期]
**上下文：** [简要描述此模式适用的场景]

## 问题
[此模式解决的具体问题 - 请详细说明]

## 解决方案
[模式/技术/变通方案 - 附带代码示例]

## 何时使用
[触发条件]
```

5. **质量门 — 检查清单 + 整体评估**

   ### 5a. 必需检查清单（通过实际阅读文件验证）

   在评估草稿前，**必须**执行以下所有步骤：

   * \[ ] 使用关键词搜索 `~/.claude/skills/` 及相关项目 `.claude/skills/` 文件，检查内容重叠
   * \[ ] 检查 MEMORY.md（项目级和全局级）是否存在重叠
   * \[ ] 考虑是否追加到现有技能即可满足需求
   * \[ ] 确认这是一个可复用的模式，而非一次性修复

   ### 5b. 整体评估

   综合检查清单结果与草稿质量，选择**以下之一**：

   | 评估结果 | 含义 | 下一步操作 |
   |---------|---------|-------------|
   | **保存** | 独特、具体、范围明确 | 进入步骤 6 |
   | **改进后保存** | 有价值但需完善 | 列出改进项 → 修订 → 重新评估（仅一次） |
   | **归入 \[X]** | 应追加到现有技能 | 显示目标技能及新增内容 → 步骤 6 |
   | **放弃** | 琐碎、冗余或过于抽象 | 说明理由并停止 |

**评估维度**（用于辅助判断，不计分）：

* **具体性与可操作性**：包含可直接使用的代码示例或命令
* **范围适配性**：名称、触发条件与内容保持一致，聚焦于单一模式
* **独特性**：提供现有技能未覆盖的价值（依据检查清单结果）
* **可复用性**：未来会话中存在合理的触发场景

6. **评估结果对应的确认流程**

* **改进后保存**：展示所需改进项 + 修订后的草稿 + 更新后的检查清单/评估结果（仅一次重新评估）；若修订后的评估结果为**保存**，则在用户确认后保存，否则按新评估结果执行
* **保存**：展示保存路径 + 检查清单结果 + 一行评估理由 + 完整草稿 → 用户确认后保存
* **归入 \[X]**：展示目标路径 + 新增内容（diff 格式） + 检查清单结果 + 评估理由 → 用户确认后追加
* **放弃**：仅展示检查清单结果与理由（无需确认）

7. 保存 / 归入至指定位置

## 步骤 5 的输出格式

```
### 检查清单
- [x] skills/ grep: 无重叠 (或: 发现重叠 → 详情)
- [x] MEMORY.md: 无重叠 (或: 发现重叠 → 详情)
- [x] 现有技能追加: 新文件合适 (或: 应追加到 [X])
- [x] 可复用性: 已确认 (或: 一次性 → 丢弃)

### 裁决: 保存 / 改进后保存 / 吸收到 [X] / 丢弃

**理由:** (用 1-2 句话解释裁决)
```

## 设计原理

此版本用基于清单的整体裁决系统取代了之前的 5 维度数字评分标准（具体性、可操作性、范围契合度、非冗余性、覆盖度，评分 1-5）。现代前沿模型（Opus 4.6+）具有强大的情境判断能力 —— 将丰富的定性信号强行压缩为数字评分会丢失细微差别，并可能产生误导性的总分。整体方法让模型自然地权衡所有因素，产生更准确的保存/放弃决策，同时明确的清单确保不会跳过任何关键检查。

## 注意事项

* 不要提取琐碎的修复（拼写错误、简单的语法错误）
* 不要提取一次性问题（特定的 API 中断等）
* 专注于那些将在未来会话中节省时间的模式
* 保持技能聚焦 —— 每个技能一个模式
* 当裁决为“吸收”时，追加到现有技能，而不是创建新文件
