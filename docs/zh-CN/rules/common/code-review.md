# 代码审查标准

## 目的

代码审查确保代码在合并前具备质量、安全性和可维护性。本规则定义了代码审查的时机与执行方式。

## 审查时机

**必须触发审查的情形：**

* 编写或修改代码后
* 提交至共享分支前
* 修改安全敏感代码时（认证、支付、用户数据）
* 进行架构变更时
* 合并拉取请求前

**审查前要求：**

在请求审查前，请确保：

* 所有自动化检查（CI/CD）均通过
* 合并冲突已解决
* 分支与目标分支保持同步

## 审查清单

在标记代码完成前：

* \[ ] 代码可读性强且命名规范
* \[ ] 函数功能聚焦（<50行）
* \[ ] 文件内聚性强（<800行）
* \[ ] 无深层嵌套（>4层）
* \[ ] 错误处理明确
* \[ ] 无硬编码密钥或凭据
* \[ ] 无console.log或调试语句
* \[ ] 新功能包含测试
* \[ ] 测试覆盖率不低于80%

## 安全审查触发条件

**遇到以下情况时，立即停止并使用安全审查代理：**

* 认证或授权代码
* 用户输入处理
* 数据库查询
* 文件系统操作
* 外部API调用
* 加密操作
* 支付或财务代码

## 审查严重等级

| 等级 | 含义 | 操作 |
|-------|---------|--------|
| 严重 | 安全漏洞或数据丢失风险 | **阻止** - 合并前必须修复 |
| 高 | 错误或重大质量问题 | **警告** - 建议合并前修复 |
| 中 | 可维护性问题 | **提示** - 考虑修复 |
| 低 | 风格或次要建议 | **备注** - 可选 |

## 代理使用

使用以下代理进行代码审查：

| 代理 | 用途 |
|-------|---------|
| **代码审查代理** | 通用代码质量、模式、最佳实践 |
| **安全审查代理** | 安全漏洞、OWASP Top 10 |
| **TypeScript审查代理** | TypeScript/JavaScript特定问题 |
| **Python审查代理** | Python特定问题 |
| **Go审查代理** | Go特定问题 |
| **Rust审查代理** | Rust特定问题 |

## 审查工作流

```
1. 运行 git diff 了解变更
2. 首先检查安全清单
3. 审查代码质量清单
4. 运行相关测试
5. 验证覆盖率 >= 80%
6. 使用合适的代理进行详细审查
```

## 常见问题排查

### 安全

* 硬编码凭据（API密钥、密码、令牌）
* SQL注入（查询中的字符串拼接）
* XSS漏洞（未转义的用户输入）
* 路径遍历（未清理的文件路径）
* 缺少CSRF保护
* 认证绕过

### 代码质量

* 大型函数（>50行） - 拆分为更小函数
* 大型文件（>800行） - 提取模块
* 深层嵌套（>4层） - 使用提前返回
* 缺少错误处理 - 明确处理
* 可变模式 - 优先使用不可变操作
* 缺少测试 - 增加测试覆盖率

### 性能

* N+1查询 - 使用JOIN或批处理
* 缺少分页 - 在查询中添加LIMIT
* 无界查询 - 添加约束
* 缺少缓存 - 缓存高开销操作

## 批准标准

* **批准**：无严重或高等级问题
* **警告**：仅存在高等级问题（谨慎合并）
* **阻止**：发现严重问题
