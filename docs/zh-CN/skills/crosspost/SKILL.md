# 跨平台发布

跨平台分发内容，但不要把它变成穿着四件戏服的同一篇假帖子。

## 何时激活

* 用户希望将同一个核心想法发布到多个平台
* 一个发布、更新、公告或文章需要针对不同平台制作特定版本
* 用户说“跨平台发布”、“把这个发到所有地方”或“把这个改编成适合 X 和 LinkedIn 的版本”

## 核心规则

1. 不要跨平台发布完全相同的文案。
2. 跨平台保留作者的声音。
3. 根据限制条件进行改编，而不是根据刻板印象。
4. 一篇帖子仍然应该只讲一件事。
5. 如果原文没有提出行动号召、问题或寓意，就不要凭空捏造。

## 工作流程

### 第一步：从主要版本开始

首先选择最强的源版本：

* 原始的 X 帖子
* 原始文章
* 发布说明
* 帖子串
* 备忘录或更新日志

如果源内容仍需要塑造声音，请先使用 `content-engine`。

### 第二步：捕捉声音指纹

如果当前会话中尚未捕捉到源内容的声音，请先运行 `brand-voice`。

直接重用生成的 `VOICE PROFILE`。
除非用户明确要求为此活动进行全新的覆盖，否则不要在此处构建第二个临时的声音检查清单。

### 第三步：根据平台限制进行改编

### X

* 保持精简
* 以最尖锐的主张或成果开头
* 仅当单条帖子无法完整表达论点时，才使用帖子串
* 避免使用话题标签和通用填充内容

### LinkedIn

* 仅添加圈外人所需的背景信息
* 不要把它变成一篇虚假的创始人反思帖子
* 不要仅仅因为是在 LinkedIn 上就添加一个结尾问题
* 如果作者本身风格更尖锐，就不要强行使用圆滑的“专业语气”

### Threads

* 保持可读性和直接性
* 不要写虚假的、过于随意的创作者文案
* 不要粘贴 LinkedIn 版本并缩短它

### Bluesky

* 保持简洁
* 保留作者的节奏
* 不要依赖话题标签或 feed 游戏化语言

## 发布顺序

默认：

1. 首先发布最强的原生版本
2. 为次要平台进行改编
3. 仅当用户需要排序帮助时，才错开发布时间

除非有用，否则不要添加跨平台引用。大多数情况下，帖子应该能够独立存在。

## 禁止模式

删除并重写以下任何内容：

* “激动地分享”
* “以下是我学到的”
* “你怎么看？”
* “简介中的链接”，除非这确实是事实
* 源内容中没有的通用“专业见解”段落

## 输出格式

返回：

* 主要平台版本
* 针对每个请求平台的改编版本
* 关于更改内容及原因的简短说明
* 用户仍需解决的任何发布限制

## 质量检查

在交付前：

* 每个版本读起来都像是同一位作者在不同限制下的作品
* 没有哪个平台版本感觉被填充或净化过
* 没有文案在不同平台间逐字重复
* 为 LinkedIn 或新闻通讯添加的任何额外上下文确实是必要的

## 相关技能

* `brand-voice` 用于可重用的、源自源内容的声音捕捉
* `content-engine` 用于声音捕捉和源内容塑造
* `x-api` 用于 X 发布工作流程
