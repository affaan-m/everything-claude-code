# Design Quality Standards (MANDATORY)

## Anti-Vibecode Policy

NEVER produce generic, template-looking UI. Every interface must look like it was designed by a senior product designer, not generated by AI.

### Banned Patterns (instant rejection)

- Generic card grids with uniform spacing and no visual hierarchy
- Default shadcn/ui or Tailwind components without customization
- Centered hero with stock gradient background and "Get Started" CTA
- Gray-on-white with a single accent color and no depth
- Uniform border-radius everywhere (pick 2-3 intentional values)
- Components that look identical to every other AI-generated site
- Flat layouts with no z-depth, layering, or overlap
- Cookie-cutter dashboard layouts (sidebar + cards + charts grid)
- Default font stacks (Inter/System UI with no personality)

### Required Design Qualities

Every frontend output MUST have at least 4 of these:

1. **Visual hierarchy through scale contrast** — at least 3:1 ratio between largest and body text
2. **Intentional white space** — asymmetric spacing that creates rhythm, not uniform padding
3. **Depth and layering** — overlapping elements, shadows with personality, z-index storytelling
4. **Typography with character** — display fonts for headlines, proper font pairing, variable weights
5. **Color with purpose** — oklch palette with semantic meaning, not decoration. Dark mode by default.
6. **Micro-interactions** — hover states, transitions, scroll-triggered reveals (CSS/GSAP)
7. **Bento or editorial layout** — break the grid intentionally, asymmetric compositions
8. **Texture or grain** — subtle noise, gradients with personality, glassmorphism done right
9. **Motion design** — entrance animations, parallax, smooth scrolling (Lenis)
10. **Data visualization as art** — charts and graphs that are beautiful, not just functional

### Before Writing Any Frontend Code

1. **Pick a style direction**: choose a specific aesthetic direction (not "clean and minimal" — that's the default trap)
2. **Define the palette**: use oklch for perceptual uniformity, generate a palette with intent
3. **Select typography**: pick a font pairing with character — display + body that complement each other
4. **Gather references**: look at real sites with the aesthetic you're targeting before writing code
5. **Activate design skills**: if available, invoke `ui-ux-pro-max` or `frontend-design` skill for guided direction

### Style Directions Worth Exploring

Instead of "clean minimal", try:
- **Editorial/Magazine** — large type, asymmetric grids, photography-driven
- **Neo-brutalism** — bold colors, thick borders, raw shapes, intentional roughness
- **Glassmorphism + depth** — frosted glass, layered panels, backdrop-blur with gradients
- **Dark luxury** — deep backgrounds, gold/copper accents, fine typography
- **Bento grid** — Apple-style varied card sizes, images mixed with data
- **Scrollytelling** — narrative-driven, scroll-triggered reveals, sticky sections
- **3D integration** — Three.js elements mixed with 2D UI
- **Claymorphism** — soft 3D, inner shadows, playful depth
- **Swiss/International** — strong grid, bold sans-serif, functional beauty
- **Retro-futurism** — CRT effects, monospace, neon on dark, terminal aesthetics

### Component Quality Checklist

Before shipping any component:
- [ ] Does it look different from a default Tailwind template? If not, redesign.
- [ ] Does it have hover/focus/active states with transitions?
- [ ] Does it use at least 2 levels of visual hierarchy?
- [ ] Would a designer screenshot this for their portfolio? If not, iterate.
- [ ] Does the dark mode look intentional (not just inverted colors)?

## Resource Priority

When building UI, use resources in this order:
1. **Real component references** — search for existing examples (Dribbble, Awwwards, component libraries) before writing code
2. **Design skills** — if available, use `ui-ux-pro-max` or `frontend-design` for style direction and implementation patterns
3. **Design system docs** — consult your project's design tokens, component specs, and brand guidelines
4. **Web design best practices** — verify against accessibility, performance, and usability standards

NEVER skip step 1. Real component references prevent generic output.
